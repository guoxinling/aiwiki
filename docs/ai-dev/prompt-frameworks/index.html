🏗️ 提示词框架

📌 概述​

提示词框架是经过验证的提示词结构模板。使用框架，可以让你不需要每次都从零开始构思提示词，直接套用结构就能写出高质量的提示词。

框架的作用就像写作大纲或PPT模板——帮你理清思路，确保不遗漏关键信息。

🎯 为什么使用框架？​

1. 提高效率​

  • 不用每次重新思考"该怎么问"
  • 直接填空式使用，3分钟完成复杂提示词

2. 保证质量​

  • 经过验证的结构，涵盖关键要素
  • 避免遗漏重要信息

3. 便于复用​

  • 一次学习，多次使用
  • 可以积累自己的框架库

🛠️ 常用框架介绍​

框架1：RTF框架（Role-Task-Format）​

最简单的框架，适合大部分场景。

结构：

【角色(Role)】你是一位...
【任务(Task)】请帮我...
【格式(Format)】请以...形式输出

示例：

角色：你是一位资深营养师
任务：为一位想减肥的上班族设计一周晚餐食谱
格式：用表格形式，包含菜名、主要食材、热量

框架2：CO-STAR框架​

新加坡政府科技局推荐的开源框架，非常系统化。

结构：

【C-Context】背景：提供任务背景信息
【O-Objective】目标：明确要完成的任务
【S-Style】风格：指定写作风格
【T-Tone】语气：设定情感基调
【A-Audience】受众：说明目标读者
【R-Response】回复：指定输出格式

示例：

背景：我是一家电商公司的运营经理，准备双11大促
目标：写一封给全体员工的动员邮件
风格：激励人心、数据驱动
语气：热情、鼓舞人心
受众：公司全体员工，包括技术和业务团队
回复：邮件格式，500字左右，分3个部分

框架3：BROKE框架​

专为解决复杂问题设计，强调背景和目标。

结构：

【B-Background】背景：阐述问题背景
【R-Role】角色：设定AI身份
【O-Objectives】目标：明确任务目标
【K-Key Results】关键结果：定义成功标准
【E-Experiment】试验：要求改进和优化

示例：

背景：我们的产品最近用户流失率上升，从5%上升到12%
角色：你是一位用户增长专家
目标：分析可能的原因并提出解决方案
关键结果：列出3个最可能的原因，每个原因配1个具体解决方案
试验：如果方案不够具体，请补充实施步骤

框架4：APE框架（Action-Purpose-Expectation）​

强调行动和预期，适合执行类任务。

结构：

【Action】行动：具体要做什么
【Purpose】目的：为什么要做
【Expectation】预期：期望的结果是什么

示例：

行动：分析这份销售数据，找出增长最快的产品品类
目的：为下季度采购决策提供依据
预期：输出Top 3品类，每类包含增长率、主要驱动因素、下季度建议

框架5：TAG框架（Task-Action-Goal）​

最简化的框架，适合快速使用。

结构：

【Task】任务：要完成的任务
【Action】行动：具体行动步骤
【Goal】目标：最终目标

示例：

任务：准备一场30分钟的产品培训
行动：设计培训大纲，包含产品介绍、核心功能演示、常见问题
目标：让新员工能独立向客户介绍产品

💡 框架选择指南​

场景推荐框架原因
快速问答RTF简单直接
正式文档CO-STAR全面系统
问题解决BROKE强调分析和优化
执行任务APE目标导向
日常琐事TAG极简快速

📝 实践案例​

案例1：写产品需求文档​

使用CO-STAR框架：

背景：我们要开发一款面向大学生的记账App，竞品是随手记、挖财
目标：写一份产品需求文档的"功能模块"章节
风格：专业、结构清晰、技术可实现性强
语气：客观、理性
受众：技术团队和产品经理
回复：分功能模块列出，每个模块包含功能描述、用户价值、优先级（P0/P1/P2）

案例2：准备面试回答​

使用BROKE框架：

背景：我明天要面试产品经理岗位，可能会被问到"你如何处理需求变更"
角色：你是一位有10年经验的互联网大厂产品总监
目标：帮我准备这个问题的回答
关键结果：回答要体现专业性，包含具体案例，时长控制在2分钟内
试验：如果回答太理论化，请补充一个具体的工作场景例子

案例3：写小红书文案​

使用RTF框架：

角色：你是一位资深小红书博主，擅长生活方式类内容
任务：为一款便携咖啡机写种草文案
格式：小红书风格，带emoji，300字以内，包含标题、正文、标签

🔧 进阶技巧​

1. 组合使用框架​

复杂任务可以组合多个框架。比如先用BROKE分析问题，再用CO-STAR输出报告。

2. 创建自己的框架​

使用一段时间后，你会发现某些固定模式。把它们总结成自己的框架，效率更高。

3. 框架不是死的​

根据情况灵活调整。如果某个要素不重要，可以省略。

❓ 常见问题​

Q1: 必须严格按照框架来写吗？

A: 不用。框架是指导，不是约束。熟练后可以灵活调整，甚至只用框架的一部分。

Q2: 同一个任务可以用不同框架吗？

A: 可以！不同的框架会带来不同的视角和输出。可以尝试对比效果。

Q3: 框架之间有什么区别？

A: 主要是侧重点不同：

  • RTF：简单直接
  • CO-STAR：全面系统
  • BROKE：问题解决
  • APE：执行导向

Q4: 新手从哪个框架开始？

A: 建议从 RTF框架 开始，最简单。熟练后尝试 CO-STAR，最全面。
